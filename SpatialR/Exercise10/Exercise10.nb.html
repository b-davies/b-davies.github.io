Exercise 10:

Spatial analysis with raster objects

This exercise is based on a study of ecological niche modeling techniques, but applied to data of Bigfoot sightings:

Lozier, J. D., P. Aniello, and M. J. Hickerson. 2009. “Predicting the Distribution of Sasquatch in Western North America: Anything Goes with Ecological Niche Modelling.” Journal of Biogeography 36 (9): 1623–27. https://doi.org/10.1111/j.1365-2699.2009.02152.x.

It also draws on materials from the originators of the terra package. Here, we will use Bigfoot sightings and environmental data to evaluate what conditions would be conducive for finding Bigfoot.

Get background values

In order to make predictions from raster data about what conditions we might expect Bigfoot to inhabit, we ideally want points that indicate presence and absence. However, the sample we have is not systematic, something that is also true for many datasets in the anthropological sciences.

Instead, we can assess the sightings data against background variability. This is similar to using CSR in point pattern studies. This will compare environments where Bigfoot has been seen to what it would be like if Bigfoots (Bigfeet?) were literally everywhere.

bgEnv<-spatSample(envData,5000,"random",na.rm=TRUE, xy=TRUE)

Creating the training dataset

To create the training data for our model, we’ll subset our data to the western USA. We could model the entire country, but like many species in North America, there is likely to be distinct Bigfoot subspecies between the east and west. We’ll use 102 W as our dividing line:

#western bigfoot points
bigfootWest<-subset(bfEnv,x < -102 )

For the model data, we need to know whether an observation is positive (1) or negative (0) as well as the associated environemental data (in this case rain and temperature). To do this we’ll want to know how many positive and negative cases we have. We can use length for this on the first column of each dataset:

#Get number of observations in the positive and background classes
bfObs<-length(bigfootWest[,1])
bgObs<-length(bgEnv[,1])

Next, we’ll use append to create vectors for rain and temperature with the bigfoot observations first, then the background observations following:


rainfall<-append(bigfootWest$rain,bgEnv$rain)
temperature<-append(bigfootWest$temp,bgEnv$temp)

Then we create a

pa<-c(rep(1,bfObs),rep(0,bgObs))

Finally, we bring it all together into a single dataframe:

bfData<-data.frame(pa,rainfall,temperature)

head(bfData)

Now we’re ready to start model-building!

Using randomforest to build a model

Random Forest is a machine learning algorithm based on decision trees. Decision-trees go through a sequence of yes/no questions to determine when a condition is true or false. Think about a decision about whether to go out to a restaurant. Do you have food you should eat at home? Are you able to afford it? And so on. The decision tree structure is used to find an optimal solution, but when applied to a dataset, it suffers from overfitting.

Random forests alleviate this by generating many random combinations of decision trees on subsets of the data and takes the average or majority vote on the . It’s a reasonably efficient approach to regression/classification problems can improve the generalizability of your model.

This approach is well-suited to use more than two variables, but for simplicity’s sake we’re just using rain and temperature.

Test data

It’s good practice to break down your model into a test and training dataset for cross-validation. Here, we’ll remove 20% of the data as a test dataset:

#Select a set of 20% rows from the data
i <- sample(nrow(bfData), 0.2 * nrow(bfData))
#Make that the test data
bfTest <- bfData[i,]
#Make the rest the training data
bfTrain <- bfData[-i,]

TWe won’t be getting into the ins and outs of cross-validation with spatial data in this exercise, but if you’re interested in learning more about this stage of model-building, you can find more at Robin Lovelace’s Geocomputation with R.

Running randomForest

Before we get going, we need to make sure R treats our positive/negative indicator as a factor rather than a number. We can do that using as.factor:

#Treat pa as a factor
fpa <- as.factor(bfTrain[, 'pa'])

Now that’s sorted, so we can run the model using randomForest.

bfRF <- randomForest(bfTrain[, 2:ncol(bfTrain)], fpa)

As written here, the randomforest function is taking only two arguments: the explanatory variables bfTrain[, 2:3] which subsets the training dataset to the last two columns, and then the factored positive/negative indicator. There are additional attributes you can modify, including the number of trees and

A couple of things to note here: the model has an OOB, or “Out of the Bag”, error rate of 14.62%, which means that it generally fails about 15% of the time. That’s not too bad. The “Confusion matrix” tells us that, on the whole, it mispredicts a background point about 6% of the time, but mispredicts a Bigfoot point about 53% of the time. This is in part due to disparities in our data (we have a lot more background than we do Bigfoot).

We can look at the relative importance of our variables using the varImpPlot function:

varImpPlot(bfRF)

If we were using more parameters to build our model, then we could try and tune the random forest model using the tuneRF and try to get a better fit. But with two parameters this isn’t likely to buy us much of an improvement, and THE HUNT FOR BIGFOOT CANNOT WAIT A MOMENT LONGER.

Spatial prediction

We can use terra’s predict function to apply our model to

Let’s take a look at where we are 85% likely to find bigfoot in the western continental USA.

names(envData)<-c("rainfall","temperature")
bigfootHabitat <- predict(envData, bfRF, na.rm=TRUE)

And now we can plot the result:

e <- ext(xmin(bigfootHabitat), -102, ymin(bigfootHabitat), ymax(bigfootHabitat))
bigfootHabitat2<-crop(bigfootHabitat,e)
plot(bigfootHabitat2)

And there you have it! If you’re looking for Bigfoot, our analysis suggests your best bet would be to go looking in the Pacific Coast ranges or perhaps in the Rocky Mountains.

At this point we could use the test data we saved to cross-validate our model; the packages predicts or mlr3viz can be used for this. This exercise demonstrates how we can use this approach to predictive modeling in an cryptoecological context, but hopefully its clear how this would be applied in other pursuits as well.
